# servicenow-coder-global

**Global-scope changes for ServiceNow** — the gated counterpart to
[`servicenow-coder`](https://github.com/orwellsanimal/servicenow-coder).

Where `servicenow-coder` ships scoped apps through fully-automated CI/CD,
this repo ships **global-scope changes through human-gated update sets**.
Each change is an AI-generated, deterministic artifact reviewed in a pull
request and committed to the target ServiceNow instance by a human.

This is the deliberate gating tier of the agentic SDLC.

## Why a separate repo?

| Concern | Scoped apps (`servicenow-coder`) | Global changes (this repo) |
|---------|----------------------------------|----------------------------|
| Deploy mechanism | `now-sdk install` via CI/CD | Update set import via Studio |
| Approval flow | Tests pass → auto-merge | CAB / human review → human commit |
| Audit trail | Git history + ATF results | Git history + `sys_update_xml` |
| Blast radius | One scoped namespace | Cross-cutting platform behavior |
| Rollback | `now-sdk rollback <context>` | Update set back-out |

Mixing these would obscure the scoped-app pipeline. Keeping them separate
respects the different risk models.

## How a change flows

```
1.  AI generates manifest.yaml + source/ in changes/<date>-<name>/
        │
        ▼
2.  Run the generator (lives in servicenow-coder):
    node ../servicenow-coder/scripts/build-update-set.js
         changes/<date>-<name>/
        │
        ▼
3.  built/update-set.xml + impact.md + preview.md + sha256 committed
        │
        ▼
4.  Pull request opened. Reviewer:
    ─ reads impact.md and preview.md
    ─ checks the source/ files
    ─ verifies update-set.sha256 matches the XML
    ─ optionally runs the live-instance preview tool
        │
        ▼
5.  PR approved. CAB / change-management ticket if required.
        │
        ▼
6.  Developer downloads built/update-set.xml, imports via:
    ServiceNow → System Update Sets → Retrieved Update Sets
    → Import Update Set from XML
        │
        ▼
7.  Preview Update Set on the instance. Compare against preview.md.
        │
        ▼
8.  Commit Update Set on the instance. Drop a `.applied` marker file
    in the change folder and merge the PR closing the change.
        │
        ▼
9.  Post-import verification:
    python ../servicenow-coder/scripts/python/preview-update-set.py
       changes/<date>-<name>/built/update-set.xml --post-import
```

See [`docs/gating-procedure.md`](docs/gating-procedure.md) for the full procedure.

## Repository structure

```
servicenow-coder-global/
├── changes/                          # one folder per change
│   └── <date>-<name>/
│       ├── manifest.yaml             # change description + artifact list
│       ├── source/                   # source files (BR scripts, etc.)
│       ├── built/                    # generated by the generator
│       │   ├── update-set.xml        # importable artifact
│       │   ├── update-set.sha256     # integrity hash
│       │   ├── impact.md             # impact summary
│       │   └── preview.md            # pre-import preview
│       ├── README.md                 # change-specific runbook
│       └── .applied                  # marker file once committed
├── docs/
│   ├── gating-procedure.md           # standard review + commit procedure
│   └── manifest-schema.md            # manifest format reference
├── .github/
│   └── workflows/
│       └── validate.yml              # CI: hash check, XML well-formedness
└── README.md                         # this file
```

## Manifest format (quick reference)

```yaml
name: Add u_needs_field_service field to incident_task
description: |
  Required for the FSM bridge scoped app integration.
application: global
target_release: 2026-Q2

intent: |
  Enable analysts to flag an incident task as needing field service.

artifacts:
  - type: dictionary_entry
    table: incident_task
    element: u_needs_field_service
    column_label: Needs Field Service
    internal_type: boolean
    default_value: 'false'

  - type: business_rule
    table: incident_task
    name: FSM Bridge - Dispatch on Flag
    when: after
    action_insert: true
    action_update: true
    condition: current.u_needs_field_service == true
    script_file: source/dispatch-br.js

estimated_impact:
  tables_modified: [incident_task]
  rollback_complexity: low
```

Full schema: [`docs/manifest-schema.md`](docs/manifest-schema.md).

## Supported artifact types

**Phase 1** (today):
- `dictionary_entry` — `sys_dictionary` rows for custom fields on existing tables
- `business_rule` — `sys_script` rows for global-scope Business Rules

**Phase 2** (planned):
- `db_object` — new tables (`sys_db_object`)
- `choice` — choice values (`sys_choice`)
- `ui_policy` — UI Policies (`sys_ui_policy`)
- `ui_action` — UI Actions (`sys_ui_action`)
- `script_include` — global-scope Script Includes

## Code Signing (enterprise extension)

For environments with ServiceNow Vault + Code Signing enabled, the generated
update set XML can be cryptographically signed before being committed to the
protected instance. The Phase 1 SHA-256 hash provides tamper evidence; Code
Signing adds cryptographic non-repudiation. The structural model is the same:

- `servicenow-docs/markdown/platform-security/sign-specific-records.md`
- `servicenow-docs/markdown/platform-security/code-signing-landing.md`

Signing slots between PR approval and import.
